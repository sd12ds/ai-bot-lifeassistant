# Conversational Nutrition Engine v2.5

## 1. Проблема и цель

### 1.1. Что ломается сейчас
Пользователь «Любимая» (230124634) выявила системные проблемы:
- Фото еды обрабатывается **отдельно** от диалогового агента — контекст теряется
- Caption и Vision не сливаются в одну сущность → бот создал 3 отдельных meal вместо 1
- Vision проигнорировал текст пользователя (50г → 150г)
- Бот не понимает подтверждения («занёс?», «сохранил?») — отвечает «уточните что именно»
- Редактирование граммовок — мучительный FSM, «10 грам» отклонено
- Бот не проявляет инициативу: не спрашивает подтверждение, не предлагает корректировку

### 1.2. Корень проблемы — разделённая архитектура
Архитектура чата разбита на событийные острова: photo handler, text handler, supervisor, nutrition agent, FSM — каждый изолирован. Нет **единого диалогового состояния**. Пользователь ведёт одну беседу, а бот воспринимает её как 3–5 несвязанных микропроцессов.

### 1.3. Цель апгрейда
Превратить Nutrition-модуль из «трекера с распознаванием» в **Conversational Nutrition Engine** — интерактивный движок, который:
- Понимает еду как **непрерывный диалог**, а не разовые команды
- Объединяет фото, текст, уточнения и правки в единый сценарий
- Не заставляет пользователя страдать от ручного FSM-редактирования
- Может быть использован как **универсальный паттерн** chat-driven управления для fitness, tasks, habits, reminders, coaching, business-модулей

## 2. Новый продуктовый принцип

**Питание = живой режим взаимодействия, а не функция**

Пользователь не «вызывает функцию логирования». Он разговаривает с ассистентом, а система сама:
- распознаёт намерение
- объединяет данные из разных сообщений
- понимает, что относится к одному приёму пищи
- предлагает подтверждение
- вносит запись
- позволяет быстро поправить
- строит выводы, рекомендации и follow-up

Примеры кросс-доменных сценариев:
- «Я выпил 300 мл воды» → лог воды
- «Добавь ещё банан» → редактирование последнего meal
- «Покажи сколько осталось по белку» → аналитика
- «Сделай мне ужин на 600 ккал» → рекомендация
- «Завтра напомни поужинать до 20:00» → мост nutrition + reminders + calendar
- «После тренировки подсказывай белок» → мост nutrition + fitness
- «Собери недельный отчёт» → мост nutrition + coaching / dashboard

## 3. Архитектура: 5 слоёв Conversational Nutrition Engine

### 3.1. Input Understanding Layer
Принимает любые типы входа: текст, фото, фото+caption, голос→текст, короткие фразы, команды-подтверждения, фразы-корректировки.

### 3.2. Session Context Layer
Хранит «о чём сейчас разговор»:
- текущий pending meal draft
- распознанные продукты + источник данных (vision/text/mixed)
- ожидается ли подтверждение
- можно ли трактовать следующее сообщение как правку
- какой последний meal сохранён
- какой шаблон поведения у пользователя повторяется

### 3.3. Intent Resolution Layer
Определяет что хочет пользователь: создать новый приём, изменить последний, подтвердить, удалить, уточнить порцию, добавить воду, спросить статистику, получить рекомендацию, создать шаблон, использовать meal как базу.

### 3.4. Action Layer
Вызывает tools / storage (полный набор описан в разделе 9).

### 3.5. Response Orchestration Layer
Формирует живой, инициативный ответ: краткое резюме, что бот понял, что сохранил, что нужно подтвердить, быстрые кнопки, следующая лучшая операция.

## 4. Универсальный Chat Action Core

Стратегически — не просто улучшить питание, а создать **общий механизм chat-state-driven управления**, который переиспользуется в других разделах.

### 4.1. Новый базовый слой: `bot/core/chat_action_engine.py`
Задача: обрабатывать любую сущность бота через единый шаблон — распознавание, определение активного объекта, хранение pending draft, подтверждение, редактирование, follow-up, откат/отмена, человекоподобный UX.

Универсальная схема обработки:
User message → classifier/supervisor → domain agent → action resolver → session context → domain tools → response composer

Для Nutrition это станет **первой полноценной реализацией**. Потом тот же паттерн — для Fitness, Tasks, Coaching, CRM.

### 4.2. Session Context Manager: `bot/core/session_context_manager.py`
Хранит:
- `active_domain` — какой модуль сейчас активен
- `current_draft` — текущий черновик сущности
- `last_saved_entity` — последняя сохранённая сущность
- `pending_confirmation` — ждём ли подтверждения
- `last_intent` — последний intent пользователя
- `editable_target` — что можно редактировать
- `source_payloads` — данные из разных источников (vision, caption, text)
- `expires_at` — таймаут сессии

В контексте Nutrition: pending meal, last saved meal, possible correction target, message chain map.

### 4.3. Chat Action Resolver
Слой, который решает: **это новая запись или правка старой?**

Пример: после фото пользователь пишет «50 г». Без resolver бот теряется. С resolver бот понимает:
- это не новый meal
- это правка последнего uncertain item
- если uncertain item один — обновить его
- если items несколько — спросить, к чему относится

### 4.4. Entity Draft Lifecycle (универсальный)
`created → enriched → awaiting_confirmation → confirmed → saved → editable_recently → archived`

Этот lifecycle применим для любых сущностей: meals, workouts, tasks, habits.

### 4.5. Дополнительные core-компоненты
- `bot/core/correction_parser.py` — разбор свободных корректировок
- `bot/core/followup_suggester.py` — движок follow-up рекомендаций

## 5. Пользовательские сценарии

### 5.1. Фото + подпись
Пользователь отправляет фото + «50 г овсянки, ложка топинга, 1 хлебец и 3 кусочка сыра».
Бот: распознаёт фото → учитывает caption как источник приоритета → объединяет в один meal draft → показывает что понял, суммарные ккал/БЖУ, сомнительные пункты → спрашивает «Всё верно? Сохранить? Поправить порции?»

### 5.2. Быстрая правка естественным языком
«овсянка 50г», «сыра не 50, а 30», «убери хлебец» — бот понимает это как **редактирование текущего pending meal**, не новый запрос. Без FSM «введите число» — естественный слой разбора.

### 5.3. Контроль сохранения
«занес?», «сохранил?», «записал это?» — бот проверяет pending state / last saved meal и отвечает по делу:
- «Да, записала: овсянка, сыр, хлебец — 260 ккал»
- «Пока не сохранила, жду твоего подтверждения»

### 5.4. Повторяющиеся приёмы
Если пользователь регулярно ест похожее — бот предлагает: сохранить шаблон, повторить прошлый приём, «как вчера», «мой обычный завтрак».

## 6. Новые фичи

### 6.1. Pending Meal Draft
После распознавания создаётся **не сразу final meal**, а `pending_meal_draft`:
- draft_id, список items, confidence по каждому item
- источник: photo | caption | text | mixed
- статус: draft → awaiting_confirmation → confirmed → saved → discarded
- last_user_message_id, created_at, updated_at

Пока пользователь уточняет — бот работает с draft, не плодит лишние meals.

### 6.2. Smart Merge Engine
Механизм слияния данных из vision, caption, follow-up text, history, food dictionary.
Правила приоритета:
1. Пользовательская правка > всего
2. Caption > Vision по граммовкам
3. Если продукт похож по названию — fuzzy merge
4. Если система сомневается — пометить item как uncertain

### 6.3. Confidence-based ответы
- Уверен: «Записала завтрак: овсянка 50г, сыр 30г…»
- Сомневается: «Похоже, здесь овсянка, сыр и хлебец. По сыру не до конца уверена — оставляем так или подправим?»

### 6.4. Inline Quick Actions 2.0
Расширенный набор кнопок: Сохранить, Исправить, Добавить/Убрать продукт, Повторить как шаблон, Поставить время, Изменить тип приёма, Показать остаток по целям, Сохранить как «мой обычный завтрак».

### 6.5. Естественные команды-корректировки
Свободный язык: «банан 120», «сделай 80 г», «овсянки меньше», «убери сыр», «добавь чай без сахара», «это был перекус, а не завтрак», «сохрани как ужин», «время 21:30».

### 6.6. Follow-up Suggestions Engine
После сохранения бот ведёт дальше:
- «До цели по белку осталось 48 г»
- «Воды сегодня ещё мало, добавить 250 мл?»
- «Хочешь сохранить этот завтрак как шаблон?»
- «Сегодня уже 80% калорий закрыто, на ужин лучше лёгкий вариант»

### 6.7. Personal Nutrition Memory
Память о привычках: любимые продукты, типичные порции, повторяющиеся завтраки, частое время еды, типичные ошибки в логировании, режим похудения/набора/поддержания. Это **персонализация nutrition behavior**, а не просто память чата.

### 6.8. Smart Portion Heuristics
«сыр», «ложка пасты», «три кусочка», «полбанана», «маленькая тарелка» — бот оценивает вес по эвристикам + уточняет только если нужно.

### 6.9. Meal Templates
Шаблоны приёмов: шаблон завтрака, перекуса, post-workout meal, «как вчера». Ускоряет логирование и усиливает ощущение «бот знает меня».

### 6.10. Meal Cloning from History
«как вчера», «повтори вчерашний завтрак», «тот же ужин», «обычный кофе» — поиск похожего meal в истории + подтверждение.

### 6.11. Вода как часть общего потока
Не отдельная функция, а conversational entity: «выпил 300 мл», «добавь воды», «налей 250», «сегодня воды мало?»

### 6.12. Photo-linked meals
Привязка фото к meal: photo_file_id у Meal + snapshot-поля КБЖУ для MealItem. Даёт: красивое отображение в MiniApp, повторное открытие meal, обучение на истории.

## 7. Интерактивность и retention

### 7.1. Режимы общения
Пользователь выбирает: строгий контроль, дружелюбный помощник, минималистичный трекер, коуч по результату.

### 7.2. Daily Nutrition Score
Ежедневный расчёт: попадание в калории, баланс БЖУ, вода, регулярность, пищевая дисциплина → простой score «Сегодня 82/100».

### 7.3. Микро-геймификация
streak логирования, streak по воде, серия дней без пропусков, лучшие недели, идеальный завтрак 5 дней подряд.

### 7.4. Smart Evening Wrap-up
Вечером — итоги дня: где перебор, чего не хватило, коррекция на завтра.

### 7.5. Nutrition Coach Overlay
Не просто логировать, а помогать: «у тебя системно не добирается белок», «перекусы уводят калории в жиры», «после тренировки окно без белка».

### 7.6. Конструктор приёмов пищи
«собери завтрак на 450 ккал», «дай ужин с упором в белок», «из того, что я обычно ем» — бот как nutrition planner.

### 7.7. Связки с Fitness
До/после тренировки: рекомендация, напоминание по белку/воде, коррекция калорий в тренировочный день.

### 7.8. Связки с Calendar / Reminders
Напоминание о завтраке, окно для ужина, напоминание пить воду, привязка к режиму дня.

### 7.9. Быстрые кнопки после каждого ответа
Вода 250 мл / Повторить вчерашний завтрак / Добавить перекус / Показать остаток / Сохранить как шаблон.

### 7.10. AI-summary по неделе
Средний калораж, отклонения, лучшие/худшие дни, белок/вода/перекусы, 3 точечных рекомендации.

## 10. Модификация промпта nutrition_agent

В промпт обязательно добавить правила:
1. **Фото + подпись = один контекст** — агент трактует как единый input
2. **Приоритет пользовательских данных** — текстовая граммовка > визуальной оценки
3. **Краткая фраза после meal draft = вероятная правка** — «30 г», «сыр 20», «убери банан», «это ужин»
4. **Подтверждения по контексту** — «сохранил?», «занёс?» — проверка статуса, не новый интент
5. **Проактивность** — после распознавания/сохранения предлагать следующий логичный шаг
6. **Без технических деталей** — никаких ID, JSON, ошибок пайплайна, callback ids
7. **Стиль** — живой, короткий, человеческий, как нормальный помощник по питанию

## 13. Риски
1. **Слишком много логики в агенте** — решение: выносить правила в кодовые резолверы и tools, не перегружать промпт
2. **Vision без user override** — решение: жёстко зафиксировать приоритет текста пользователя
3. **Нет draft-сущности** — без неё снова будут плодиться лишние meals и рваться контекст
4. **Невынесенная reusable-логика** — решение: сразу проектировать общий каркас, а не «уникальный nutrition-костыль»
5. **Переусложнение UI** — нельзя превратить чат в форму; сила в том, что пользователь пишет естественно, а структура рождается внутри
